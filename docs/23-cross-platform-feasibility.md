## 0. 결론 요약

**기술적으로 가능하다. 다만 "이식"이 아니라 "UI 셸 재작성"이다.**

- **코드의 40%(16,102 LOC)는 이미 플랫폼 중립**이고, 맥에서 **오늘 그대로 빌드·테스트가 돈다**
  (147 테스트 green, 실측 §1). 파일 목록·탭·경로 바·도크 같은 **화면 로직 자체가 중립 크레이트에
  들어 있다** — 여기가 이 프로젝트의 자산이다.
- **나머지 60%(23,939 LOC)는 HWND에 붙어 있다.** win.rs(5,906)·ctl 컨트롤 18종(7,139)·
  그 위에 세워진 대화상자들(prefs 2,019·bulkrename 2,239 등)은 **이식 대상이 아니라 재작성 대상**이다.
- **비용은 "코어 재사용 + 창/입력/텍스트/셸 통합 신규 구현"**이며, 플랫폼당 **십수 K LOC 규모**로
  본다(§4). 이는 M0~M5 전체(약 3주, 40K LOC)에 비추면 **플랫폼당 M1~M3에 준하는 사업**이다.
- **가장 큰 제약은 기술이 아니라 확정 결정이다.** DR-1(관리 런타임·UI 프레임워크 금지)·
  DR-8(외부 crate 0 지향)·DR-2(예산)는 **Windows에 인박스 API가 있다는 전제**에서 성립했다.
  Linux에는 그런 인박스 GUI가 없다 — **DR 개정 없이는 Linux 확장이 성립하지 않는다**(§3).
- **기능 패리티는 100%가 될 수 없다.** 셸 컨텍스트 메뉴(IContextMenu)는 macOS·Linux에 등가물이
  없다(§2 표의 ⚠ 항목).

**권고**: 지금 결정하지 말고 **§6의 3단계**를 밟는다 — ① 중립성 회귀 방지 규율 명문화(비용 0,
이미 CI가 절반 수행) → ② **맥 렌더 스파이크**(nexa-gui 재사용이 실제로 성립하는지 1~2주 실증,
M0-7 렌더 스파이크와 같은 방식) → ③ 그 결과로 판단. 스파이크 없이 하는 추정은 전부 추측이다.

### 1-2. 이미 맥에서 돌아가는 것 (오늘 실측)

```
cargo test -p nexa-core -p nexa-vfs -p nexa-tree -p nexa-ops -p nexa-term -p nexa-gui
→ 147 passed; 0 failed  (맥 네이티브, aarch64-apple-darwin)

cargo build -p nexa-app  → 성공(맥 스텁 바이너리 454KB — main.rs의 cfg(not(windows)) 경로)
```

- **`main.rs`에 이미 플랫폼 격리 골격이 있다**: 플랫폼 모듈은 `#[cfg(windows)]`,
  순수 로직 모듈은 `#[cfg_attr(not(windows), allow(dead_code))]`로 **전 플랫폼 컴파일**된다.
  M0 설계([docs/01](01-architecture.md) §1) 때 넣은 규율이 지금도 유지되고 있다.
- **CI가 이미 ubuntu·macOS에서 `cargo test --workspace`를 돌린다**([ci.yml](../.github/workflows/ci.yml)
  `core` job). 즉 **중립성은 이미 회귀 감시 중**이다 — 확장을 하든 안 하든 이 자산은 유지된다.

### 1-3. 결정적으로 유리한 구조 하나

화면 로직이 `nexa-app`이 아니라 **`nexa-gui`(중립)에 있다**:

| `widgets/pathbar.rs` | 715 | 경로 바(breadcrumb) |
| `widgets/chrome.rs` | 499 | 도구 모음·툴바 버튼 |

써 주면 앱의 주 화면이 그려진다** — 이것이 §6 스파이크의 근거다.

## 2. 플랫폼 결합 지점 — 무엇을 새로 써야 하는가

`nexa-app`의 Windows 결합 23,939 LOC를 성격별로 나누고, macOS·Linux 대응을 정리했다.
LOC는 **Windows 구현의 크기**이며, 대체 구현 크기는 다를 수 있다(대개 창·입력은 비슷하고,
셸 통합은 플랫폼마다 편차가 크다).

### 2-1. UI 셸 — 재작성 대상 (약 19.5K LOC)

| 모듈 | LOC | Windows 구현 | macOS | Linux | 난이도 |

> 이미 잡혀 있다 — ctl은 07-17 이후 **`DrawCtx` 백엔드로만 그리도록 규약이 개정**됐다

### 2-3. 모델 층에서 새로 생기는 문제

코드가 아니라 **개념**이 달라지는 지점 — 중립 크레이트에도 영향이 간다.

- **드라이브 vs 마운트**: X-17에서 도입한 가상 최상위 `::PC::` 센티널과 `C:\` 절대 경로 모델은
  Windows 드라이브 개념에 맞춰져 있다. macOS는 `/Volumes`, Linux는 `/media`·`/mnt`·`/run/media`로
  **최상위 개념 자체를 재해석**해야 한다(`nexa-tree`/`nexa-vfs` 영향).
- **경로 의미론**: 경로 구분자, 대소문자 구분(APFS 기본 비구분 / ext4 구분 / NTFS 비구분),
  심볼릭 링크·하드링크, 파일명 허용 문자, 숨김 규칙(`.`접두 vs 속성 비트).
  → **X-13 대소문자 정렬 규약**(07-15 확정)은 플랫폼별로 다시 정의해야 한다.
- **글리프 자산**: 내비·디스클로저 화살표를 **Segoe MDL2 Assets**로 통일했는데(07-18)
  이 폰트는 Windows 전용이다. **다행히 대체 경로가 이미 있다** — 07-18에 만든
  **SVG 서브셋 파서(`svg.rs` 679 LOC, 중립·crate 0)**로 아이콘을 그리고 있으므로,
  MDL2 글리프도 SVG 자산으로 옮기면 플랫폼 독립이 된다.
- **UI 관례**: 메뉴 바 위치(맥 = 화면 상단 글로벌 메뉴), 단축키 수식자(Ctrl↔⌘),
  창 버튼 배치, 스크롤 방향·오버레이 스크롤바(맥 관례는 이미 채택 07-18).

## 3. 확정 결정과의 충돌 (여기가 진짜 관문)

| 결정 | 내용 | macOS | Linux |
| --- | --- | --- | --- |
| **DR-1** | 올 러스트 단일 바이너리 · **관리 런타임·UI 프레임워크·WebView 금지** · 커스텀 드로잉 | 조건부 성립 — AppKit은 OS 인박스지만 **Objective-C 런타임 경유**가 불가피(`objc2`류 crate 또는 직접 `objc_msgSend`) | **성립 불가(현행 문안대로면)** — 인박스 GUI 자체가 없어 X11/Wayland·폰트 스택을 **외부에서 가져와야** 함 |
| **DR-2** | 유휴 RSS ≤30MB · exe ≤10MB · **임포트 = OS 인박스 DLL만** | 인박스 프레임워크 링크로 대응 가능(측정 필요) | **"인박스" 개념이 없음** — 게이트 재정의 필요. 폰트 스택 포함 시 exe·RSS 증가 불가피 |
| **DR-6** | PolyForm NC + 의존성 **퍼미시브 온리(GPL 금지)** | ✅ `objc2`·`core-foundation-sys` 등 MIT/Apache | ⚠ **GTK(LGPL)·Qt(GPL/상용) 배제.** 단 X11·Wayland·libxkbcommon·HarfBuzz = MIT, FreeType = FTL 선택 시 퍼미시브 → **원시 계층으로 가면 준수 가능** |
| **DR-8** | 외부 crate 기본 0 지향(현재 승인 = `windows`·`windows-core`·`regex-lite`) | 개정 필요(플랫폼 바인딩 crate 추가) | **대폭 개정 필요**(창·입력·폰트·셰이핑) |

**해석**: DR-1의 취지는 "관리 런타임과 UI 프레임워크에 종속되지 말고 우리가 직접 그린다"였다.
**커스텀 드로잉이라는 본질은 크로스 플랫폼에서도 유지 가능**하다 — 바뀌는 것은 "Win32가
공짜로 주던 창·컨트롤·텍스트를 어디서 조달하느냐"뿐이다. 그러나 **문안 그대로는 Linux에서
위반이 되므로, 확장을 결정하는 순간 ADR-0005(가칭)로 DR-1·DR-2·DR-8을 함께 개정해야 한다.**
([16 §2 규칙 5](16-doc-git-conventions.md) — 과거를 지우지 않고 새 항목으로 정정)

| 리스크 | 없음 | **높음** — Wayland/X11 이중 대응·IME·HiDPI·a11y를 전부 자력 | 중간 — crate 생태계 의존·퍼미시브 확인 필요(winit Apache-2.0·softbuffer/cosmic-text MIT/Apache = ✅) | 낮음 |
| 소요(감) | — | 플랫폼당 **M1~M3급 사업** | 플랫폼당 **M1급 + 셸 통합** | 소 |

> **소요는 감(感)이다.** 근거 있는 수치는 §6 스파이크 이후에만 나온다. 참고 좌표:
> Windows 구현은 M0~M5 전체가 약 3주(07-11~07-15 집중 + 이후 UX 고도화)였고 그 산출이 40K LOC다.

**현실적 후보는 C다.** 커스텀 드로잉 아키텍처(=이 프로젝트의 정체성)를 유지하면서 창·입력·
텍스트라는 "OS가 공짜로 주지 않는 부분"만 퍼미시브 crate로 조달하는 형태이고, **`DrawCtx`
트레이트가 이미 그 접합면으로 존재**한다. B는 정합성은 최고지만 Wayland·X11·IME·a11y를
자력으로 감당해야 해서 이 규모의 프로젝트에는 과대하다.

## 5. 결정하려면 필요한 것

확장을 **결정**한다면 다음이 같은 시점에 필요하다.

1. **ADR-0005 — 크로스 플랫폼 확장**: DR-1(문안)·DR-2(예산 정의)·DR-8(crate 원장) 개정.
   대안 B/C/D 비교와 채택 근거를 남긴다.
2. **예산 재정의**: "임포트 = OS 인박스 DLL만"(B3)의 플랫폼별 등가 규칙. exe·RSS 게이트를
   플랫폼별로 따로 둘지 결정.
3. **패리티 정책**: 셸 컨텍스트 메뉴처럼 **대응물이 없는 기능을 어떻게 처리할지**
   (기능 축소 / 자체 메뉴 대체 / 플랫폼별 기능 표 공개). [19 패리티 갭](19-parity-gap.md) 방식 확장.
4. **배포 채널 설계**: macOS = `.app` 번들 + dmg + **공증(notarization, Apple 개발자 계정 필요)** —
   현재 **무서명 방침**(DR-3)과 정면 충돌한다. Linux = AppImage/deb/rpm/Flatpak 중 선택.
   [21 배포](21-distribution.md) §7·§8의 winget·choco 심사가 아직 진행 중임도 고려.
5. **QA 환경**: 현재 실행 검증 신뢰 원천은 Windows CI다([11](11-dev-environment.md)).
   플랫폼이 늘면 **실기 QA 부담이 3배**가 된다 — 지금도 실기 QA가 병목인 점을 감안해야 한다.

## 6. 권고 — 결정 전에 밟을 3단계

**지금 확장을 결정할 필요는 없다.** 아래는 결정을 미루면서도 선택지를 살려 두는 순서다.

### 1단계 — 중립성 회귀 방지 (비용 ≈ 0, 지금 바로)

- 이미 CI `core` job이 ubuntu·macOS에서 `cargo test --workspace`를 돌린다 → **현상 유지가 곧 자산 보존**.
- 규율 한 줄만 추가한다: **"새 로직은 우선 중립 크레이트에 놓고, 플랫폼 API는 `nexa-app`
  경계 안에서만 호출한다"**([15 개발 방법론](15-dev-methodology.md)에 편입 후보).
- 효과: 확장을 안 하더라도 **테스트 가능성·구조 위생**이 올라간다. 손해가 없다.

### 2단계 — 맥 렌더 스파이크 (1~2주 규모, M0-7과 동일 방식)

**검증 질문 하나**: "`nexa-gui`의 `rows.rs`(2,885 LOC)를 손대지 않고 맥 창에 파일 목록을 그릴 수 있는가?"

- 범위: 창 1개 + 이벤트 루프 + `DrawCtx` 구현 1개(텍스트·사각형·클립) + 실제 디렉터리 열거.
  **선택·편집·대화상자·셸 통합은 전부 제외.**
- 측정: 첫 렌더 시간(P1 대응)·스크롤 프레임(P2)·유휴 RSS(B1)·바이너리 크기(B2) —
  Windows 실측치와 같은 축으로 비교([STATUS §2](STATUS.md)).
- 산출: **중립 크레이트 수정량**(0에 가까울수록 설계 검증 성공) + 실제 소요 시간 →
  이것으로 §4의 "감"을 수치로 바꾼다.
- 실패해도 얻는 것: `DrawCtx` 추상화의 실제 누수 지점 목록 = Windows 코드 품질에도 도움.

### 3단계 — 판단

스파이크 결과로 ① 중립 크레이트 수정량 ② 소요 ③ 예산 실측을 확보한 뒤,
§5의 5개 항목을 놓고 A/C/D 중 선택한다. **macOS 먼저, Linux는 그다음**을 권한다 —
맥은 인박스 프레임워크가 있어 DR 충돌이 작고, Linux는 Wayland/X11 이중 대응이라 난이도가
한 단계 높다.

## 7. 미해결 질문 (사용자 결정 필요)

1. **동기가 무엇인가** — 사용자 저변 확대? 개발자 본인의 맥 상시 사용? 판매(ctl 라이브러리
   포함) 확대? **동기에 따라 D(코어 재사용)만으로 충분할 수도 있다.**
2. **어느 플랫폼이 먼저인가** — macOS만으로 충분한가, Linux까지 필요한가.
3. **패리티를 어디까지 포기할 수 있는가** — 셸 컨텍스트 메뉴 없는 파일 탐색기를 같은 제품으로 볼 것인가.
4. **무서명 방침(DR-3)을 유지할 것인가** — macOS 배포는 공증 없이는 사실상 불가능하다.
5. **실기 QA 3배 부담을 감당할 체제가 있는가** — 현재도 QA는 사용자 1인이 수행 중이다.
