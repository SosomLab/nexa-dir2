  원본 동일 수용) ④ 클래식 룩 이질감 수용(원본 사용자 결정 계승).

## M3-3 Undo/Redo — OperationHistory 이식·휴지통 복원 (`feat/m3-undo`)

> 원본 이식: `Nexa.ViewModels/OperationHistory.cs`(+ `OperationHistoryTests.cs`) — B-13u,
> docs/33 §Undo/Redo. 휴지통 복원은 `Nexa.App/RecycleBin.cs`(S2 — 셸 undelete 동사).

- **nexa-ops `history` 모듈**(플랫폼 중립): `ReversibleOp` trait + `OperationHistory`(undo/redo
  스택 2개·새 push 시 redo 무효화·상한 100·**실패 연산은 스택에서 소실**=무결성 우선, 원본 규약).
  연산 4종 — `MoveBatchOp`(역이동)·`CopyBatchOp`(사본 삭제 주입)·`RenameOp`(new↔old)·
  `CreateOp`(생성물 삭제 주입/재생성 주입). 오류는 **`OpError`로 구조화**(i18n 문구는 앱 책임 —
  원본의 Localizer 직접 호출을 계층 분리). 테스트 9 이식(스택 규약 5 + 연산 왕복 4).
- **앱 배선**(win.rs): `State.history`(세션 한정) + push 3곳 — **전송 완료**(Move/CopyBatchOp,
  `Outcome.transferred` 수행 쌍만·취소돼도 수행분 기록)·**이름변경**(RenameOp, 동일 이름 무동작
  제외)·**새로 만들기**(CreateOp, undo=휴지통·redo=재생성 주입). **Ctrl+Z=실행 취소·Ctrl+Y/
  Ctrl+Shift+Z=다시 실행** + **편집 메뉴**(Undo/Redo). 실행 후 항상 양쪽 재로드·결과는 타이틀
  노트(성공=설명/실패=OpError→i18n 사유). i18n en/ko: undo/redo·history.*·op.*Count·rename.done.
- **휴지통 복원**(`recycle.rs` — 원본 RecycleBin.cs): 휴지통 셸 폴더 열거 → "원래 위치+이름" 매칭
  (정확 일치·확장자 숨김 폴백) → 셸 **`undelete` 동사**(탐색기 Ctrl+Z와 동일). COM 초기화 국소
  수행·PIDL/STRRET 수동 해제. **원본과 차이**: `StrRetToBufW`(shlwapi) 대신 STRRET 직접 파싱 —
  **신규 임포트 DLL 0 목표**(단, `ole32.dll`은 CoInitialize/CoTaskMemFree로 신규 — OS 인박스라
  DR-2 준수, B3 화이트리스트 등재). `DeleteBatchOp`(앱 계층 — 셸 COM 의존): undo=복원·redo=재삭제,
  Del(휴지통) 경로에 push. **완전 삭제는 undo 불가**(설계상 제외 — 확인창 방어, 원본 동일).
- **검증**: 워크스페이스 test 전부 green(ops 20·history 9 포함) · **실 휴지통 왕복 통합 테스트**
  (삭제→복원→원위치·내용 무손상, `#[ignore]` 수동) 로컬 통과 · release exe **0.58MB**(B2) ·
  기동 RSS **25.51MB**(B1) · **B3 통과**(ole32 등재).
- **α 한계(후속)**: 편집 메뉴 Undo/Redo **활성/비활성 표시 없음**(위젯 enabled 미지원 — 없으면
  타이틀 노트로 알림) · 다중 버전 휴지통 항목은 경로당 최초 일치 1건(삭제 시각 비교=후속) ·
