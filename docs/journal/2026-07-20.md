### X-25 — 일괄 이름변경 조상·자손 동시 선택 처리 (결함 확인)

- **코드 확인 결과 결함 재현**: 순수 변환 로직([batch_rename.rs](../crates/nexa-ops/src/batch_rename.rs))은
  각 항목 독립 처리라 미리보기까지 정상. 문제는 **적용 단계**
  ([win.rs `open_bulk_rename`](../crates/nexa-app/src/win.rs)).
- 적용 루프는 다이얼로그 열 때 캡처한 절대 경로(`old_path`)들을 순차
  `nexa_ops::rename`(= `old_path.parent().join(new_name)`) 한다. **부모 폴더를
  먼저 개명하면 자손의 저장된 `old_path`가 디스크에서 소멸** → 자손 rename 실패.
- 선택 순서 = [`keyboard_targets`](../crates/nexa-app/src/win.rs) →
  `selected_paths()` = **삽입 순서(비결정적)**. 부모 우선 시 자손 **실패**(오류 집계),
  자손 우선 시 겉보기 성공이나 done 쌍이 stale → **undo**([`MoveBatchOp::move_all`](../crates/nexa-ops/src/history.rs)
  reverse에서 `exists(from)` false로 건너뜀)·[`rename_expanded`](../crates/nexa-app/src/win.rs)
  경로 추적 손상.
- [`conflicts()`](../crates/nexa-ops/src/batch_rename.rs)는 부모 문자열 동등만 봐
  이 조상-자손 관계를 감지 못함(부모 폴더의 부모 = `C:\`, 자손의 부모 = `C:\proj` — 서로 다름).
- **대응 결정(사용자)**: 1차 = **감지 후 차단**(선택 집합의 조상-자손 관계 검출 →
  자손을 새 `Conflict` 종류[예: `Nested`]로 표기·적용 제외·미리보기 사유 표시).
  2차(차후) = **경로 rebase 완전 지원**(얕은 것부터 적용 + 각 rename 성공 후 남은
  pairs `old_path` 접두사 치환[`rename_expanded` 재사용]·실제 결과 경로로 done 기록 →
  undo 정합). TODO §7 X-25 등록.
