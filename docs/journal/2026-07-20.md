### X-28 — 탭 우클릭 New Tab + 멀티라인 탭·상하 드래그

- **New Tab 메뉴**: [win.rs `show_tab_menu`](../crates/nexa-app/src/win.rs)에
  `CMD_NEW` 추가 — 잠금/고정/복제 아래·**탭 닫기 바로 위**(사용자 지정 위치).
  동작 = Ctrl+T·[+] 동일 경로(`new_tab` — 활성 탭 경로 복제). i18n `tab.new`(ko/en).
- **멀티라인 탭**([tabbar.rs](../crates/nexa-gui/src/widgets/tabbar.rs) 개편):
  폭 초과 시 다음 줄로 줄바꿈(줄 첫 탭은 예외 없이 배치·[+]도 동일 흐름).
  히트 캐시 = x 범위 → **탭별 Rect**(줄 반영). 한 줄 높이 = `set_line_height`
  (호스트 = `PanelMetrics.tab_h`).
- **줄 수 산정은 paint**(텍스트 측정은 paint에서만 — edit.rs 캐시 규약):
  `lines()` 캐시 + `take_lines_changed()` 1회성 표지 → [win.rs `paint`](../crates/nexa-app/src/win.rs)
  말미에서 수거해 **재레이아웃 + 무효화**(다음 프레임 수렴 — 1프레임 지연 허용).
  [panel.rs `set_bounds`](../crates/nexa-app/src/panel.rs) = 탭 바 높이 `tab_h × lines`.
- **상하 드래그 이동**: 드래그 임계 8px를 세로에도 적용, 스냅 규칙은 기존
  "대상 탭 x 중간점 통과"를 그대로 일반화 — 줄바꿈 흐름상 **아래 줄 = 항상 뒤
  인덱스**라 `(to > from && x ≥ mid) || (to < from && x ≤ mid)` 규칙이 교차 줄에도
  성립(오실레이션 없음). 테스트: 줄바꿈 줄 수·교차 줄 히트·상하 Move 3종 추가.

### X-29 — 도크 텍스트 문자 단위 선택 복사 (미리보기 보완)

- 기존(QA 07-15)은 **라인 단위** 선택 — 사용자 요청 "드래그로 선택된 영역 복사"에
  맞춰 [dock.rs](../crates/nexa-gui/src/widgets/dock.rs)를 **문자 단위**로 보완
  (정보/미리보기 텍스트 공통·이미지/터미널 종류는 종전대로 제외).
- 선택 모델 = (라인, 문자 경계) 앵커→현재. **문자 경계 x 오프셋은 paint가 캐시**
  (edit.rs `paint_field` 접두 폭 규약 — 폭 초과 문자는 측정 중단 = 상한).
  내용/지표/크기 변경 시 캐시 무효 → 다음 paint 재측정(매 프레임 재측정 없음).
- 드래그 = 영역 하이라이트(라인별 부분 fill + `ctx.text` 겹쳐 그리기 — 편집 필드
  규약), 라인 경계 밖은 첫/끝 라인·경계로 클램프. **이동 없는 단순 클릭 = 선택
  해제**(edit.rs 규약 — 종전 "클릭 = 라인 선택"에서 변경). Ctrl+C 경로
  (`selected_text` — 다중 라인 = `\r\n` 결합)는 호스트 무변.

### X-25 1차 — 일괄 이름변경 조상·자손 동시 선택 차단 ✅

- [batch_rename.rs `Conflict::Nested`](../crates/nexa-ops/src/batch_rename.rs) 신설 —
  `conflicts()`가 선택 집합 안에서 **개명 예정(new≠old)인 다른 항목의 전체 경로
  (부모+현재 이름)가 이 항목 부모의 경로 접두**이면 자손을 Nested로 표기(경로 계층
  비교 — 대소문자·구분자 정규화, 접두 유사 형제 `parents` 오탐 없음. 조상이
  무변경이면 자손 안전 = 충돌 아님).
- 다이얼로그([bulkrename.rs](../crates/nexa-app/src/bulkrename.rs))는 기존 충돌
  경로 그대로 — 미리보기 사유 `bulk.conflict.nested`("상위 폴더와 함께 선택됨")
  표시 + 적용 버튼 차단. 테스트 `nested_selection_blocks_descendants` 추가.
- 2차(경로 rebase 완전 지원 — 얕은 것부터 적용·접두 치환·undo 정합)는 후속 유지.

### X-25 — 일괄 이름변경 조상·자손 동시 선택 처리 (결함 확인)

- **코드 확인 결과 결함 재현**: 순수 변환 로직([batch_rename.rs](../crates/nexa-ops/src/batch_rename.rs))은
  각 항목 독립 처리라 미리보기까지 정상. 문제는 **적용 단계**
  ([win.rs `open_bulk_rename`](../crates/nexa-app/src/win.rs)).
- 적용 루프는 다이얼로그 열 때 캡처한 절대 경로(`old_path`)들을 순차
  `nexa_ops::rename`(= `old_path.parent().join(new_name)`) 한다. **부모 폴더를
  먼저 개명하면 자손의 저장된 `old_path`가 디스크에서 소멸** → 자손 rename 실패.
- 선택 순서 = [`keyboard_targets`](../crates/nexa-app/src/win.rs) →
  `selected_paths()` = **삽입 순서(비결정적)**. 부모 우선 시 자손 **실패**(오류 집계),
  자손 우선 시 겉보기 성공이나 done 쌍이 stale → **undo**([`MoveBatchOp::move_all`](../crates/nexa-ops/src/history.rs)
  reverse에서 `exists(from)` false로 건너뜀)·[`rename_expanded`](../crates/nexa-app/src/win.rs)
  경로 추적 손상.
- [`conflicts()`](../crates/nexa-ops/src/batch_rename.rs)는 부모 문자열 동등만 봐
  이 조상-자손 관계를 감지 못함(부모 폴더의 부모 = `C:\`, 자손의 부모 = `C:\proj` — 서로 다름).
- **대응 결정(사용자)**: 1차 = **감지 후 차단**(선택 집합의 조상-자손 관계 검출 →
  자손을 새 `Conflict` 종류[예: `Nested`]로 표기·적용 제외·미리보기 사유 표시).
  2차(차후) = **경로 rebase 완전 지원**(얕은 것부터 적용 + 각 rename 성공 후 남은
  pairs `old_path` 접두사 치환[`rename_expanded` 재사용]·실제 결과 경로로 done 기록 →
  undo 정합). TODO §7 X-25 등록.
