## 문서·커밋/푸시 규약 문서화 — 타 프로젝트 이식용 (사용자 요청)

- **배경**: 사용자가 "다른 프로젝트에도 동일 적용" 목적으로 현행 문서 관리 기준과
  commit/push 기준의 요약을 요청 → 구두 요약에 그치지 않고 **붙여넣어 지시할 수 있는
  규약 문서**로 고정.
- **신설** [16 문서·커밋/푸시 규약](../16-doc-git-conventions.md):
  - **§0 지시문 블록** — 새 프로젝트 AI 세션에 그대로 붙여넣는 코드펜스(문서 체계·커밋·
    브랜치/푸시 3부). 문서 본체를 읽히지 않아도 규약이 전달되도록 자족적으로 작성.
  - **§1 문서 4층 체계** — 진입(CLAUDE.md)/현황(STATUS·MILESTONES·TODO)/경과(DEVLOG·
    journal·BRANCHES)/지식(NN·ADR) + README 색인. 설계 원리 = **시간축·목적축·상태축 분리**,
    상세는 journal 한 곳이고 나머지는 요약+링크.
  - **§2 작성 필수 규칙 8** — journal이 원본 / 전 기록 시간 역순 / 한 작업 = 한 트랜잭션
    갱신(커밋→journal→DEVLOG→MILESTONES·TODO→BRANCHES) / SSOT 동시 갱신 / 결정은 DR·ADR로
    고정(정정은 새 항목) / "왜 + 실측값" / 기존 자산 우선·출처 명기 / 문서 번호 불변.
  - **§3 커밋** — Conventional Commits + scope 규약 + **맥락 태그**(사용자 요청/QA/확정·
    백로그 ID) + main 항상 green. 본 저장소 실제 커밋 4건을 예시로 인용.
  - **§4 브랜치·푸시(🔴 필수)** — push는 명시 요청 시에만 · **태그 push 별도 승인**(공개
    Release 생성) · 파괴적 작업 사전 확인 · 그 외 자동 진행(상태 md는 무프롬프트) ·
    settings.json 병합만.
  - **§5 릴리스** 4단계(승격 커밋→승인 태그→결과 동기→외부 심사 대기 시 보류 방침 명시)
    · **§6 새 프로젝트 적용 체크리스트**(파일 골격 + 착수 4단계).
